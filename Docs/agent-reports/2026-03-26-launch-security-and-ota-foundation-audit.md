## Summary

This audit establishes the first explicit launch-readiness baseline for AddOne. The current repo has a real app distribution path and a real hosted realtime path, but it is still operating with beta and factory assumptions rather than customer-launch trust guarantees.

The highest-risk gaps are transport trust on firmware, broker and device isolation, staging-grade auth and environment posture, and the absence of a real firmware OTA system. The recommended order is to harden transport and identity first, then clean release operations, then build OTA and minimum user-facing update status.

## Decisions / Assumptions

- Customer launch should require real firmware OTA; wider beta may temporarily tolerate manual firmware distribution only for a tightly controlled cohort.
- Store-managed app updates are sufficient as the first app-update policy; Expo JS OTA is optional and not required to begin launch hardening.
- OTA and security should be treated as one connected launch workstream, not as separate implementation tracks.

## Open Questions Or Blockers

- Whether to require a separate hosted production Supabase project before wider beta, or only before customer launch.
- How strict the production broker identity model should be: one credential per device, one credential per cohort, or another scoped model.
- Whether app-side update status should stay minimal for launch or include user-controlled firmware update actions in the first launch version.
