Target launch trust model
- Firmware HTTPS trust: primary path should be CA-based certificate validation, not `setInsecure()`. For a small team, the simplest launch-grade model is to embed the needed CA roots or broker CA certificates in firmware and use `WiFiClientSecure::setCACert(...)` for Supabase and MQTT broker connections. This tolerates normal leaf-cert rotation better than leaf fingerprint pinning. Optional later hardening: add SPKI or leaf pinning only for especially sensitive endpoints such as a future OTA artifact host.
- Firmware MQTT trust: launch should require `mqtts` only, with certificate validation enabled and `kMqttAllowInsecureTls = false` in shipped configs. The current self-signed bootstrap mode can remain a lab-only emergency path, not a beta or launch trust model. [deploy/beta-vps/README.md](/Users/viktor/Desktop/DevProjects/Codex/AddOne/deploy/beta-vps/README.md#L17)
- Broker credential scope: recommended primary path is one dedicated gateway broker account plus one unique broker credential per device. The device broker secret should be separate from `device_auth_token`. `device_auth_token` remains product-auth material; MQTT credentials become transport-auth material. That separation matches the transport doc’s own stated direction and keeps future rotation sane. [AddOne_Device_Realtime_Transport.md](/Users/viktor/Desktop/DevProjects/Codex/AddOne/Docs/AddOne_Device_Realtime_Transport.md#L149)
- Topic ACL isolation: keep the existing topic structure, but enforce it. A device should be allowed to subscribe only to `addone/device/<hardware_uid>/command` and publish only `ack`, `presence`, `event/day-state`, `event/friend-celebration-ready`, and `snapshot/runtime` for its own `hardware_uid`. The gateway account should publish command topics and subscribe only to the device-originated wildcards it needs. Device-to-device publish or subscribe permissions should be impossible.
- Gateway privilege boundary: for the first launch-grade slice, the gateway can remain a trusted backend component with `SUPABASE_SERVICE_ROLE_KEY`, because that is the smallest operable model in this repo. The hardening work should be to keep one gateway instance per environment, one dedicated broker service account, no public write API beyond health, and no dependence on shared fleet credentials. Optional later hardening: narrow the Supabase privileges behind dedicated internal RPCs or a narrower backend role.
- Device identity lifecycle: `hardware_uid` stays the stable physical identifier. `device_auth_token` becomes a per-ownership-epoch product secret generated on-device, hashed server-side, bound during onboarding claim or factory-only service-role registration, used for HTTPS cloud RPCs, cleared on full factory reset or account removal, and regenerated on the next onboarding cycle. Field devices should not self-register through a public anon RPC.
- MQTT payload identity: recommended primary path for this first hardening slice is to keep `device_auth_token` in MQTT payloads as defense in depth, because it minimizes gateway and backend churn while transport auth is being fixed. Optional later hardening: remove `device_auth_token` from MQTT payloads after the gateway has a clean internal auth path that trusts broker-enforced identity instead of product-auth material in every message.
- Credential provisioning: recommended primary path for customer launch is dynamic per-device MQTT credential issuance after successful onboarding claim or through a dedicated authenticated device RPC, then local persistence in firmware preferences. Interim wider-beta fallback, if needed, is factory-injected per-device MQTT credentials during ship-ready flashing, but that is less flexible for rotation after shipment.

Implementation order
1. Lock the identity rules first. Decide and document that `device_auth_token` is product auth only, MQTT credentials are transport auth only, `register_factory_device(...)` is factory-only, and field token binding happens during onboarding claim or a dedicated authenticated device RPC. This removes the current ambiguity before code changes start.
2. Implement firmware CA validation for Supabase HTTPS and MQTT. This is the first code slice because every later trust decision, including OTA, is unsafe if the device still accepts any certificate.
3. Add per-device MQTT credential issuance and local persistence. Recommended primary path: issue MQTT credentials after claim and store them in firmware preferences alongside existing device-local state. Small controlled beta fallback: inject them at factory only if needed to avoid blocking early validation.
4. Add broker ACLs that enforce the existing topic map and split gateway from device credentials. Keep wildcard subscriptions only on the privileged gateway account, not on device accounts.
5. Align reset and recovery semantics. Make sure account removal and factory reset continue to clear both local and server-side device-auth state, define what happens on device auth failure, and remove any reliance on anon-key runtime self-registration.
6. Hand off OTA work only after steps 1 through 5 are settled. OTA can then build on validated HTTPS, device-isolated broker transport, and a clean device-identity lifecycle instead of inheriting the current mixed trust model.

OTA dependencies
- Firmware OTA artifact and manifest download cannot be considered safe until the firmware has a real CA-validated HTTPS stack. The current `setInsecure()` posture would make OTA unsafe by construction. [cloud_client.cpp](/Users/viktor/Desktop/DevProjects/Codex/AddOne/firmware/src/cloud_client.cpp#L497)
- Any OTA trigger or notification path that uses MQTT depends on device-isolated broker credentials and ACLs. Without that, one compromised broker credential can interfere with multiple devices’ update control plane.
- OTA should not reuse `device_auth_token` as an artifact-download credential. If `device_auth_token` stays in MQTT payloads for the first launch slice, treat that as temporary defense in depth only. OTA artifact access should later rely on signed manifests or a separate download-auth design, not on the long-lived product-auth secret.
- OTA rollback and recovery semantics depend on the identity lifecycle being clear. If a device is removed from an account or factory-reset, the system needs an explicit rule for whether prior transport credentials and future OTA eligibility are revoked immediately. The current reset flows already clear product-auth state, which OTA should inherit, not bypass.
